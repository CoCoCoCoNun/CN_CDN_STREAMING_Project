1. Demo란 내가 구현한 기능을 **보여주기** 위한 것. (의도적으로 상황을 만들기)
2. 청자의 입장에서 발표를 준비할 것. (Edge case 처리 과정에서 멈춤, 일시정지 등)
=> input()을 넣어서 의도적으로 client <--> server의 동작을 단계별로 멈췄던 것처럼, Streaming에서도 의도적으로 멈추거나 혹은 화질 전환 시 멈출 수 있도록 구현해야했음. 또한, 현재 실행 중인 chunk를 시각적으로 보여주는 장치를 더 크게 생성해야했음.
3. 보고서를 작성할 때 함축적인 표현을 사용하지 말 것. => 내 머리속에 있는 내용, 숨은 전제를 비우고 보고서를 처음 받아본 입장에서 생각할 것.

### Probe 수치가 필요한 이유.

스트리밍에서 ROF를 계산하고, 임계값을 넘으면 화질을 변환한다. 
그 목적은 스트리밍이 끊기지 않되, 최대한 화질을 유지하도록 하는 것.

문제는 화질 전환 트리거 발생 시, 청크에 남은 영상 화질을 기준으로 바꾸는 게 아닌, 현재 재생중인 것을 기반으로 바꾼다는 것
=> 즉시 화질 전환을 시도하기 때문. 

화질 상향 시에는 즉시 전환을 시도하고, 화질 하향 시에는 기존 버퍼를 그대로 두고 그 다음 청크부터 요청하는 게 맞았을 거 같음.

구현할 때, 화질이 전환되고 버퍼가 하나면 ROF가 화질 하향에 가까워지는데 음.. 만약 화질이 전환되면 ROF 회복을 위해 화질 하향 임계치를 동적으로 조정하는 게 있었으면 좋았겠네.
원래 0.2 -> 화질 전환 후 3번의 probe 정도는 0.05로 임계치 설정
